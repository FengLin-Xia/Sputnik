四、星场绘制 — 当前行为摘要（web/src/starfield.ts）
------------------------------------------------
广播音符星（非恒星）
- 平面映射 + 点亮时像素十字：axisWeight(k)≈0.8*exp(-0.246*k)，R_MAX=13；指数权重 + rgbFade，无 vel 加成。
- 侧翼加厚：t1/t2 较高；k≥4 增加第三圈 ±3 侧翼（t3）；未点亮时小十字为 5×1+1×5、块 5×5。
- 全星闪烁：flicker / flickerHi / flickerTip 振幅加大（峰值可接近 1.0）。
- 四臂不同步：armN/E/S/W（0.8+0.2*sin，相位错开）× armBrightnessMul(dx,dy) 乘到 alpha；斜向取象限平均、侧翼按主横/纵轴归类。视觉上各臂「长度」随闪烁变化。
- 背景噪点略压暗以免抢戏。

项目恒星（projects.json）
- hashString32(id) + 独立 phase / freq，与别颗恒星频闪不同步；twinkle=0.87+0.13*sin，最暗约 0.74。
- 粗核十字 + 延伸臂 d=7…17，armFade 缓降；峰值 base 约 0.4+pulse*0.14；hover 约 39×39 / 43×43；命中半径约 30px。

九、Render v0.1（pipeline/render）与 backend — 对齐 RenderV01.pr.md

十、Broadcast Package v0.1（pipeline/broadcast_package）— 对齐 Broadcast_PackageV01.pr.md
------------------------------------------------------------------------------------
协议与目录

实现模块
- `pipeline/broadcast_package/schema.py`：各 JSON 最小字段与校验；`PACKAGE_VERSION`；`write_json` / `read_json`。
- `pipeline/broadcast_package/builder.py`：`build_package(PackageInputs)` —— 用 `load_score` 校验 score 与 render 一致、拷贝 score 与 WAV、写出 state/meta/playlist/fragments；默认 state/fragments/meta 辅助函数。

- `python -m pipeline.run render`：仅 score → WAV（默认写入 `broadcast/audio/rendered.wav`）。

- `python -m pipeline.run all`：先 render 再 package（单日闭环）。
- 依赖 numpy：请使用 `backend/.venv`（或已安装 `backend/requirements.txt` 的环境）从仓库根执行。

- 静态资源：`broadcast/latest/` 生成后可同步到 `web/public/broadcast/`（并 `npm run build` 使 `dist/broadcast/` 含 WAV）。

十二、Sputnik Score v0.1 协议对齐（Sputnik_Score_Spec_PrV01.md）
----------------------------------------------------------------
规范文档：根目录 `Sputnik_Score_Spec_PrV01.md`，定义 `score.json` 最小乐谱协议（bpm / bars / notes：`t,d,p,v,track`），供 render、前端星图与后续 composition 共用。

pipeline/render/schema.py
- `validate_score` / `load_score` 与规格硬约束一致：`bpm`∈[50,100]；`bars` 仅 4 或 8；每条 note：`t`≥0，`d`∈[0.25,4]（网格单位），`p`∈[48,84]，`v`∈(0,1]，`track`∈0–3；`t+d` 不超过片段总拍数；`notes` 条数 ≤64。
- 模块内导出常量名：`BPM_MIN`/`BPM_MAX`、`BARS_ALLOWED`、`PITCH_MIN`/`PITCH_MAX`、`DURATION_MIN`/`DURATION_MAX`、`MAX_NOTES`（与规格章节对应）。

示例乐谱修正（三处内容一致）
- `broadcast/latest/score.json`、`web/public/broadcast/score.json`、`pipeline/render/demo_score.json`：`p` 超上界的一条由 90 改为 83；`d` 低于建议下限的一条由 0.2 改为 0.25，以便通过上述校验。

前端（web/src/types.ts、mapNotes.ts）
- `types.ts`：增加 `SCORE_PITCH_MIN`（48）、`SCORE_PITCH_MAX`（84）；为 `Note`/`Score` 字段补充与 v0.1 一致的 JSDoc。
- `mapNotesToStarPositions`：纵轴音高归一化改为相对规格音域 [48,84]，而非本段 notes 的 min/max，与规格「星图映射稳定区间」一致。

十三、Composition v0.1（pipeline/composition）— 对齐 Composition_v0.1_PR.md
---------------------------------------------------------------------------
目标
- 从「种子 + 可选状态」经本地模型（或 mock）生成合法 `score.json`，与 `pipeline/render/schema.py`（Sputnik Score v0.1）校验一致；不接 drift、不接用户污染、不接多模型 UI。

目录与职责
- `pipeline/composition/prompt_builder.py`：`CompositionInput`（默认风格种子、`mode`/`mood`/`signal`、预留 `extra_context`）；`build_messages()` 产出 Ollama 风格 `system` + `user`（注入规格边界，引用 schema 常量）。
- `pipeline/composition/providers/mock_provider.py`：无网络，返回固定合法 JSON（与 `demo_score.json` 同形），便于无 Ollama 联调。
- `pipeline/composition/providers/ollama_provider.py`：`POST {base}/api/chat`（标准库 `urllib`），解析 `message.content`；可配置 `model`、`base_url`、`timeout`；默认基址来自环境变量 `OLLAMA_HOST`（`host:port` 或完整 `http(s)://`）。
- `pipeline/composition/composer.py`：去 markdown 代码围栏、括号匹配抽取首个 JSON 对象；`parse_score_json` → `score_from_dict`；失败重试；`write_score_json` 写出格式化 JSON。
- `pipeline/composition/providers/__init__.py`：`TextProvider` Protocol（`generate(messages) -> str`）。
- `pipeline/composition/__init__.py`：导出 `compose`、`CompositionInput`、`write_score_json` 等。
- `pipeline/composition/__main__.py`：CLI（`--provider mock|ollama`、`--model`、`--ollama-url`、`--timeout`、`--out`、`--seed` 可重复、`--mode`/`--mood`/`--signal`、`--max-retries`）。

CLI 与产物
- `python -m pipeline.composition`（默认 `--provider mock`）：写出 `composition_output/score.json`（目录含 `.gitkeep`）。
- `python -m pipeline.run compose`：同上能力，子命令挂在总管线；`--out` 默认 `composition_output/score.json`。
- 真机 Ollama：需本机 `ollama serve` 且已拉模型；CLI 的 `--ollama-url` 或环境变量 `OLLAMA_BASE_URL` 传入完整基址时覆盖默认。

与 render 的解耦（便于无 numpy 只跑生成）
- `pipeline/render/__init__.py`：仅立即导入 `schema`；`exporter`/`synth` 经 `__getattr__` 懒加载，避免 `import pipeline.render` 即依赖 numpy。
- `pipeline/run.py`：顶部不再 `import render_file_to_wav`；仅在 `render` / `all` 子命令内导入 exporter，使 `compose` 子命令可在未安装 numpy 的环境下执行。

后续衔接
- 生成物可再经 `python -m pipeline.render --score composition_output/score.json` 转 WAV，或拷贝/接入 `broadcast_package` 与 `web/public/broadcast/`（与第六节、第十节流程一致）。

十四、GitHub Pages：VITE_BASE 双路径（*.github.io 与自定义域名根）— 2026-04-20
----------------------------------------------------------------------------
问题
- 项目站 URL 为 `https://<user>.github.io/<仓库名>/`，构建需 `VITE_BASE=/<仓库名>/`；绑定自定义域名并挂在站点根时，需 `VITE_BASE=/`。原先 workflow 写死 `/Sputnik/`，换域名后资源路径易 404。

改动（.github/workflows/pages.yml）
- 增加「Resolve VITE_BASE」步骤，按优先级取值：① `workflow_dispatch` 手动输入 `vite_base`（非空时）；② 仓库 Actions 变量 `PAGES_VITE_BASE`（非空时）；③ 默认 `/${{ github.event.repository.name }}/`（与仓库名一致，无需再手改仓库名）。
- 构建日志会打印 `Resolved VITE_BASE=...`，便于核对。

使用说明
- 长期用 `*.github.io/<repo>/`：不设 `PAGES_VITE_BASE` 即可（默认即 `/<repo>/`）。
- 长期用自定义域名根：在仓库 Settings → Secrets and variables → Actions → Variables 中设置 `PAGES_VITE_BASE` = `/`。
- 临时试一次：Actions 里 Run workflow，填写 `vite_base`（如 `/`）。
