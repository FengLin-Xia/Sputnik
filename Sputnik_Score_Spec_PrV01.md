Sputnik Score Spec v0.1
1. 目标

定义 Sputnik 项目中 score.json 的最小可用协议，用于描述一段广播片段的机器可读乐谱结构。

本规范服务于三个模块的共享协作：

render：根据 score.json 渲染音频
frontend：根据 score.json 点亮平面星图
composition：后续由本地模型输出满足本规范的合法谱面

本规范不追求通用乐谱表达能力，也不追求完整作曲格式；它只服务于 Sputnik 当前阶段的广播生成、发声和可视化闭环。

2. 文件定位

score.json 不是传统五线谱，也不是 DAW 工程文件。
它是：

Sputnik 某一段广播的最小音符结构说明书。

它描述的是：

这段广播多快
这段广播多长
哪些音在什么时间出现
每个音持续多久
每个音多高、多强
每个音属于哪条广播轨道
3. 适用范围
v0.1 适用
单段广播片段
固定节拍网格
最多四条基础轨道
冷电子广播风格的最小音符结构
支持前端星图和 render 共用
v0.1 不适用
复杂节拍变化
和弦标记、调式标记、乐理分析
自动化参数曲线
复杂音色控制
多段广播编排（由 playlist.json 负责）
完整歌曲结构表达
4. 文件结构
顶层结构
{
  "bpm": 72,
  "bars": 4,
  "notes": [
    { "t": 0, "d": 1, "p": 60, "v": 0.8, "track": 0 },
    { "t": 2, "d": 1, "p": 64, "v": 0.6, "track": 1 },
    { "t": 4, "d": 2, "p": 67, "v": 0.5, "track": 0 },
    { "t": 6, "d": 0.35, "p": 79, "v": 0.4, "track": 3 }
  ]
}
5. 顶层字段定义
bpm
类型：number
含义：广播片段的速度
用途：
render 计算时间轴
frontend 推导点亮节奏
v0.1 建议范围：
50 <= bpm <= 100

说明：
v0.1 中 bpm 不是为了复杂乐理，而只是一个统一时间尺度。

bars
类型：integer
含义：广播片段总小节数
用途：
定义片段总长度
限制 note 时间范围
v0.1 建议取值：
4 或 8

说明：
v0.1 建议先固定片段长度，不支持自由结构展开。

notes
类型：array
含义：音符列表
要求：
至少可以为空数组，但正常广播片段应包含有效 note
每个元素必须是合法 note 对象
建议按 t 升序排列
6. Note 对象定义

每个 note 对象必须包含以下字段：

{ "t": 0, "d": 1, "p": 60, "v": 0.8, "track": 0 }
t（time）
类型：number
含义：音符起始位置
单位：固定网格上的位置值
用途：
render 决定 note 何时发声
frontend 决定星点何时点亮
v0.1 约束
t >= 0
建议使用固定步长
建议按离散格对齐
note 的 t 不应超出该片段总时长
建议解释

在 v0.1 中，t 先按抽象网格位置处理，不强绑定复杂乐理。
后续 render 根据 bpm + bars + grid 解释为实际秒数。

d（duration）
类型：number
含义：音符持续时长
单位：与 t 相同的固定网格单位
用途：
render 决定音符持续多久
frontend 决定星点保持点亮多久/余辉长度
v0.1 约束
d > 0
建议范围：
0.25 <= d <= 4
不建议出现过长持续值
p（pitch）
类型：integer
含义：音高
表示方式：先按 MIDI 风格整数处理
用途：
render 映射为频率
frontend 可用于星点空间分布
v0.1 约束
建议范围：
48 <= p <= 84

说明：
这不是为了乐理表达完整，而是为了保证：

声音不至于过低或过高
星图映射有稳定区间
本地 render 更容易控制
v（velocity / volume）
类型：number
含义：力度 / 相对音量
用途：
render 控制响度
frontend 控制星点亮度或权重
v0.1 约束
0 < v <= 1
建议范围：
0.2 <= v <= 0.9

说明：
v0.1 统一使用 0~1 浮点，不使用 MIDI 的 0~127。

track
类型：integer
含义：轨道编号
用途：
render 区分不同发声层
frontend 区分不同星群语义
v0.1 合法取值
0
1
2
3
7. 轨道语义定义
Track 0：Beacon / Lead

主信号轨。
表示“机器在试图唱”的主要部分。

特征：

薄
冷
有信号感
是广播中最接近“旋律”的层

前端建议：

属于主广播星层
活跃度较高
点亮时最易被感知
Track 1：Cold Drone

底层持续轨。
表示设备仍在工作的低存在感。

特征：

长音
稀
空
冷
不抢主信号

前端建议：

分布更稀疏
点亮更稳定
更像暗冷底层星带
Track 2：Static / Ghost

故障和边缘信号层。

特征：

噪声
掉渣
闪现
不稳定

前端建议：

点亮短促
可更弱、更散
更像不稳定边缘星群
Track 3：Relay / Ping

短促遥测脉冲层。

特征：

短
冷
像应答或反弹
存在感低于主信号轨

前端建议：

可更孤立
更像远端信号节点
点亮短而明显
8. 硬约束（必须满足）
顶层
必须是合法 JSON 对象
必须包含 bpm
必须包含 bars
必须包含 notes
notes
notes 必须是数组
每个 note 必须包含：
t
d
p
v
track
数值约束
bpm：建议 50~100
bars：建议 4 或 8
t >= 0
d > 0
48 <= p <= 84
0 < v <= 1
track ∈ {0,1,2,3}
数量约束（建议）
v0.1 建议单段 note 总数不超过 64
单轨 note 数建议不超过 32

说明：
这不是最终理论上限，而是 v0.1 为了：

render 不坍塌
星图不糊
广播结构保持稀疏感
9. 软约束（建议满足）

这些不一定由程序强制拦截，但应作为 composition 和后处理的生成边界。

1. 稀疏优先

Sputnik v0.1 不追求饱满编曲，留白比填满更重要。

2. 重复允许存在

可以有重复与执念，但不应演变成机械无差别堆叠。

3. 失败的歌唱优先于完整旋律

输出应更像“机器在试图歌唱”，而不是成熟作品。

4. 冷电子广播感优先于音乐性完整

结构应服务于广播感，而不是服务于传统作曲逻辑。

5. 前端可视化兼容性优先

过密、过乱、全部堆在同一区域的 note 不适合当前项目。

10. 前端兼容性约束

由于 score.json 同时服务星图可视化，需满足以下条件：

1. note 密度不能过高

避免点阵过密导致视觉糊成一片。

2. track 分布应有层次

避免所有 note 仅出现在单一轨道，导致视觉缺乏结构层级。

3. 时间推进应可感知

不能所有 note 都挤在极短时间内。

4. v 需要可用于视觉亮度权重

建议保留一定差异度，不全部写成相同值。

11. Render 兼容性约束

由于 score.json 首先服务 render，需满足：

1. 时间和时长必须可转换为样本时间轴

t / d 不能过于随意。

2. pitch 必须在可听且可控范围内

避免 render 出现极端高低频噪声。

3. track 必须可映射到固定音色层

不能引入未定义轨道编号。

4. 单段总复杂度必须有限

避免 v0.1 render 退化为混乱噪声。

12. v0.1 设计原则总结

score.json 在 v0.1 的原则是：

简单
窄
够用
能同时服务 render 和 frontend
保留后续 composition 接入空间

它不是最终完美协议，
而是 Sputnik 第一次真正发声时使用的最小广播乐谱。

13. 示例
合法示例
{
  "bpm": 72,
  "bars": 4,
  "notes": [
    { "t": 0, "d": 1, "p": 60, "v": 0.8, "track": 0 },
    { "t": 1, "d": 2, "p": 52, "v": 0.4, "track": 1 },
    { "t": 3, "d": 0.5, "p": 75, "v": 0.3, "track": 3 }
  ]
}
不推荐示例
所有 note 全在同一时间
p 超出大量区间
v 大于 1
超高 note 数密度
未定义 track = 9
只追求“满”，没有留白
