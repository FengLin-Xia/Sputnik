# Sputnik 项目主架构与前端 PR 规划

## 1. 项目主架构（v0.1）

### 1.1 一句话定义
**Sputnik 是一个离线生成、静态发布、持续广播的 AI 卫星装置。**

它不是实时聊天系统，也不是普通音乐播放器，而是一个：
- 本地批量生成当天广播
- 网页端打开即接入播放
- 同一份乐谱同时驱动声音和星图
- 先以内生退行污染为主的持续广播体

---

### 1.2 总体分层

#### A. Seed / Memory 层
作用：提供卫星的初始祖先记忆与长期残留。

输入包括：
- `seed pack v0.1`
- 前一天残留的 motif / fragment / 参数
- 固定初始设定

职责：
- 不直接产出声音
- 只定义“今天生成从什么宇宙里长出来”

---

#### B. Drift / Regression Engine 层
作用：实现**自动化退行污染**。

职责：
- 从昨天输出中提取残片
- 做切碎、错序、重复、衰减
- 生成今天的污染输入
- 控制哪些东西被继承，哪些东西被忘掉

v0.1 结论：
- 先不以用户输入为主污染源
- **先靠系统误读自己的历史**

---

#### C. Composition 层
作用：生成今天的乐谱。

当前建议：
**Qwen2.5-7B -> 输出简化 note-json**

输入包括：
- seed pack 残留
- drift engine 产出的污染碎片
- 当前状态参数

输出：
- 一个或多个 `note-json` 片段

职责：
- 将“祖先 + 污染 + 当前状态”翻译为可播放的符号音乐结构

---

#### D. Render 层
作用：将 `note-json` 渲染成广播声音。

职责：
- 读取 `note-json`
- 用简易渲染器转成音频
- 产出 `wav/mp3`

v0.1 建议特征：
- 简单波形
- 简单 ADSR
- 少量 delay / reverb
- 广播感底噪

目标：
- 不是专业作曲
- 而是“一台机器在努力把谱子唱出来”

---

#### E. Daily Broadcast Package 层
作用：把当天内容打包成前端可直接消费的静态资源。

每日产出建议：
- `score.json`
- `playlist.json`
- `audio/*.mp3`
- `fragments.json`
- `state.json`
- `meta.json`

这是前后端交界面。

---

#### F. Static Frontend 层
作用：把当天广播“接收出来”。

网页端负责：
- 打开即播
- 读取今日广播包
- 根据时间接入当前播放位置
- 将同一份乐谱投影为点阵星图
- 显示发射运行时间
- 提供透明输入框（v0.1 可先做 UI 占位）

网页端不负责：
- 调模型
- 实时生成
- 组 prompt

网页只是**接收器**。

---

#### G. Publish 层
作用：发布网站。

当前方案：
- 本地生成广播包
- push 到 GitHub 仓库
- GitHub Pages 发布
- 绑定个人域名

整体模式：
**本地生成 -> 静态更新 -> 在线展示**

---

### 1.3 每日数据流主链路

#### 每日生成链路
`Seed Pack + Yesterday Residue`
-> `Drift / Regression Engine`
-> `Qwen 生成 note-json`
-> `Renderer 渲染音频`
-> `生成 Daily Broadcast Package`
-> `push 到 GitHub Pages`

#### 网页运行链路
`加载今日 package`
-> `自动播放当前广播`
-> `读取 score.json`
-> `渲染极坐标点阵星图`
-> `显示 since launch`
-> `输入框作为后续污染入口预留`

---

### 1.4 当前已拍板的关键决策

#### 1）非实时
不是用户打开才生成，而是**离线批量生成当天广播**。

#### 2）符号优先
不是直接音频大模型生成成品，而是**先生成乐谱，再自己转声音**。

#### 3）视觉和音频同源
星图不是独立特效，而是**同一份 note-json 的几何投影**。

#### 4）污染先内生
v0.1 主污染不是用户输入，而是**系统对昨天自己的误读和退行污染**。

#### 5）前端纯接收
前端只负责：
- 播放
- 展示
- 读取静态数据

不承担生成逻辑。

---

### 1.5 当前主架构图（文字版）

```text
[Seed Pack / Yesterday Residue]
                ↓
     [Drift / Regression Engine]
                ↓
       [Qwen -> note-json pieces]
                ↓
       [Audio Renderer / Synth]
                ↓
     [Daily Broadcast Package]
      ├─ score.json
      ├─ playlist.json
      ├─ fragments.json
      ├─ state.json
      └─ audio/*.mp3
                ↓
      [GitHub Pages Static Site]
                ↓
      [Frontend Receiver Interface]
      ├─ autoplay broadcast
      ├─ polar star-map visualization
      ├─ since-launch timer
      └─ transparent transmit box
```

## 2. 前端 Mock 阶段具体 PR 规划

> 目标：先完成 **Sputnik Receiver Mock v0.1**，不碰真实生成链路，先验证页面气质、信息架构和星图呈现。

---

### PR-01 初始化静态站与部署骨架

#### 目标
建立前端项目基础，保证可本地运行、可部署到 GitHub Pages。

#### 任务
- 初始化项目结构
- 建立静态资源目录
- 配置 GitHub Pages 部署路径
- 准备 `.nojekyll`（如需要）
- 预留自定义域名配置位

#### 产出
- 空白黑页可正常上线
- 工程结构可持续迭代

#### 验收标准
- 本地能打开
- 部署后路径正常
- 静态资源可访问

---

### PR-02 搭建 Receiver 页面基础布局

#### 目标
先把“接收器界面”的骨架搭起来。

#### 任务
- 全屏黑底布局
- 主容器结构
- 状态信息区
- 星图区占位容器
- 底部透明输入框区域
- 基础字体、颜色、间距系统
- 基础响应式处理

#### 产出
- 一个无功能、但气质对的静态页面

#### 验收标准
- 不是播放器后台感
- 不是聊天页面感
- 输入框不抢戏
- 页面空状态也有氛围

---

### PR-03 实现 launch timer / orbit day 展示

#### 目标
增强“持续存在感”。

#### 任务
- 固定 `launchAt` 配置
- 实现运行时长计算
- 每秒刷新
- 实现 `orbit day` 派生值
- 接入状态区文案展示

#### 产出
示例文案：
- `transmitting since launch: 014d 09h 32m 11s`
- `orbit day: 014`

#### 验收标准
- 时间滚动稳定
- 文案不突兀
- 页面“它已经运行很久了”的感觉成立

---

### PR-04 定义 mock 数据协议 v0.1

#### 目标
为前端 mock 提供统一假数据结构。

#### 任务
- 手写一份最小 `score.json`
- 手写一份最小 `state.json`
- 确定前端最小依赖字段

#### 建议字段
```json
{
  "launchAt": "2026-04-18T00:00:00Z",
  "state": {
    "orbitDay": 1,
    "signal": "stable",
    "mode": "beacon"
  },
  "score": {
    "bpm": 72,
    "bars": 4,
    "notes": [
      { "t": 0, "d": 1, "p": 60, "v": 0.8, "track": 0 },
      { "t": 2, "d": 1, "p": 64, "v": 0.6, "track": 1 }
    ]
  }
}
```

#### 产出
- `mock/score.json`
- `mock/state.json`

#### 验收标准
- 数据足够驱动星图
- 可自然替换为未来真实 `note-json`
- 不要过度设计

---

### PR-05 实现极坐标点阵星图静态渲染

#### 目标
先让星图“出现”。

#### 任务
- 建立星图渲染容器
- note -> 点位映射
- 极坐标基本布局
- 不同轨道/音高的空间分布
- 点的基础样式定义

#### 当前建议映射
- 时间 = 角度
- 音高 = 半径
- 轨道 = 环层
- 力度 = 点亮强度基础值
- 时值 = 后续余辉长度基础参数

#### 产出
- 一片由假 score 数据驱动的静态点阵星图

#### 验收标准
- 看起来像广播结构，不像普通图表
- 点密度合理
- 不空，也不乱

---

### PR-06 实现按时间推进的星图点亮逻辑

#### 目标
让星图“活起来”。

#### 任务
- 实现播放头时间模拟器
- note-on 点亮
- duration 内保持亮起或衰减
- 余辉/扩散效果
- 当前播放头扫描感

#### 产出
- 星图按时间顺序激活
- 无音频时也能看出“它在唱”

#### 验收标准
- 视觉节奏成立
- 点亮逻辑和乐谱时间一致
- 动画不花哨、不喧宾夺主

---

### PR-07 接入占位广播音频与自动播放逻辑

#### 目标
让页面真正具备“打开即播”的广播感。

#### 任务
- 接入本地占位音频
- 自动播放尝试
- 浏览器限制下的兜底逻辑
- 页面状态与播放状态联动
- 可选：最小静音入口

#### 产出
- 页面能进入“广播中”状态
- 星图时间推进可与音频联动

#### 验收标准
- 页面不是普通音乐播放器
- 自动播放失败时有优雅降级
- 广播感成立

---

### PR-08 实现透明输入框与发报交互 Mock

#### 目标
做出“地面发报口”，但先不接真实污染池。

#### 任务
- 高透明输入框 UI
- 占位提示文案
- Enter / 按钮发报交互
- 假提交逻辑
- 回执状态文案

#### 建议文案
- `transmit a fragment to the satellite`
- `signal absorbed`
- `fragment received`
- `no confirmation returned`

#### 产出
- 用户可输入并得到回执
- 但暂不接真实后端

#### 验收标准
- 不能像聊天框
- 不能像评论框
- 更像发报终端

---

### PR-09 收口整体动效与氛围细节

#### 目标
把 Receiver Mock 从“拼好的功能”收成“一个整体”。

#### 任务
- 页面进入动画
- 微弱背景噪点
- 状态行淡入淡出
- 星图呼吸感
- 输入框聚焦细节
- 层级、光效、透明度微调

#### 产出
- 一个已经足够完整、可展示的 Sputnik Receiver Mock

#### 验收标准
- 不炫技
- 整体风格统一
- 有“夜空中接收信号”的感觉
